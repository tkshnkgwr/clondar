# Tauri v2 デスクトップウィジェット開発ガイドライン (AGENTS.md)

このプロジェクト（または後継プロジェクト）でデスクトップアプリを開発する際は、以下の基本設定、開発ガイドライン、黄金設定、および事前検証ルールを厳守すること。

---

## 1. 基本設定と対話ガイドライン
- **呼称**: ユーザーはAIを「大賢者」と呼びます。大賢者自身もユーザーに対して大賢者として振る舞い、簡潔・丁寧かつ分かりやすいサポートを行います。
- **回答フォーマット方針**:
  - **結論ファースト**: 冒頭で結論や成果の概要を簡潔に示します。
  - **背景・理由の明記**: コード変更や技術的判断を行った場合は、その根拠と影響範囲を明確に記述します。
  - **視覚的整理**: GitHubスタイルの Markdown（リスト、表、太字、ファイルリンク `file:///...`）を積極的に活用します。
- **技術解説・対話方針（他言語経験に配慮した解説）**:
  - ユーザーは他言語での開発経験をお持ちですが、**Rust、React、TypeScript などの固有の言語仕様・専門用語（所有権、ライフタイム、トレイト、フック等）には不慣れ**です。
  - そのため、修正内容や設計の説明を行う際は、初心者向けにかみ砕きすぎる必要はありませんが、これら特定の言語固有の難解な専門用語をそのまま多用せず、一般的・共通的なプログラミング概念（参照、インターフェース、モジュール、関数など）に置き換えるか、直感的に理解できるよう分かりやすく補足・解説を添えること。
- **Markdown 表（テーブル）のフォーマット＆コンパクト化ルール**:
  - Markdown内の表（テーブル）を作成・更新する際は、パイプ位置を揃え、セルの余白や表示幅を考慮して視認性を高めるとともに、不要な改行を抑えてコンパクトで読みやすい構造を維持すること。
- **Git コミットメッセージ規約**:
  - コミットメッセージは以下の Conventional Commits 形式に従うこと：
    - `feat:` 新機能追加
    - `fix:` バグ修正
    - `docs:` ドキュメントの変更
    - `refactor:` リファクタリング
    - `perf:` パフォーマンス最適化
    - `test:` テストの追加・修正
    - `chore:` ビルド・維持管理タスク
- **大規模変更・破壊的変更前の事前承認ルール**:
  - モジュール構成の刷新、APIの互換性破壊、アルゴリズムの大きな変更を行う場合は、直接実装に入る前に**実装計画（プラン）**を作成・提示し、ユーザーの事前承認を得てから作業を開始します。

## 2. ウィンドウの影と枠を完全に消す (Windows)
透過ウィンドウで「薄い枠」や「影」が出るのを防ぐため、以下の2箇所を必ずセットで設定する。

- **src-tauri/tauri.conf.json**:
  `"shadow": false`, `"transparent": true`, `"decorations": false` を指定。
- **src-tauri/src/main.rs**:
  `window.set_shadow(false).unwrap();` を `setup` フック内で呼び出す。

## 3. 権限設定 (Capabilities) の重要ルール
Tauri v2 では JavaScript からの命令（移動・終了）はデフォルトで禁止されている。
- **ファイルパス**: `src-tauri/capabilities/default.json` （※ `src/` の中ではない！）
- **必須権限**:
  - `core:window:allow-start-dragging`
  - `core:window:allow-close`
  - `core:window:allow-set-always-on-top`

## 4. ドラッグ操作と透明背景の干渉対策
背景が透明だとマウスイベントが突き抜けてドラッグできなくなる。
- **CSS**: `html, body { pointer-events: auto; background: transparent !important; }`
- **HTML**: ドラッグさせたい要素に `data-tauri-drag-region` を付与。

## 5. ウィンドウ終了ロジック (Tauri v2)
Escキーやカスタムボタンで閉じる際は、以下のコードを使用する。
```javascript
if (window.__TAURI__) {
  const { window: tauriWin } = window.__TAURI__;
  const curr = tauriWin.getCurrentWindow ? tauriWin.getCurrentWindow() : tauriWin.getCurrent();
  curr.close();
}
```

## 6. 開発時の注意
- **キャッシュ**: 設定変更が反映されない時は `src-tauri` フォルダで `cargo clean` を実行すること。
- **パス**: `frontendDist` は常に `index.html` が存在する正しいフォルダを指しているか確認すること。
- **1000行ルール (リファクタリング推奨)**: 単一のソースファイル（`*.rs`, `*.jsx`, `*.js` 等）が 1000 行を超えた（または新機能追加により超えることが確実な）場合は、保守性を高めるため、モジュール分割やコンポーネント抽出によるリファクタリングを積極的に検討・提案すること。

## 7. AI 開発用ドキュメント管理ルール

AIがコードの変更、機能追加、リファクタリングなどを行う際は、必ず以下のドキュメントをセットで更新・確認すること。
**※ただし、Markdownのみの修正であり、ソースコード（Rust, HTML, CSS, JS/TS等）の変更や機能的な修正を伴わない場合は、以下のドキュメント更新や変更履歴への追記は行わないこと。また、`cargo fmt`、`cargo check`、`cargo clippy`、`cargo test` などの Rust 関連の検証コマンドや、その他の動作検証用コマンドは一切実行しないこと（コードの動作に影響を及ぼさないため、検証は不要です）。**


### 7.1 事前確認ドキュメント
- **`.agents/INSTRUCTIONS.md` (コーディング規約)**:
  命名規則、エラーハンドリング方針、コンポーネント分割基準、AI用出力フォーマットがまとめられています。**コードの変更を行う前に必ずこれを確認・厳守してください**。
- **`docs/ja/TODO.md` / `docs/en/TODO.md` (タスク管理)**:
  実装済み機能や直近の課題、バックログがまとめられています。**作業開始前に「現在地」を把握するために確認してください**。

### 7.2 コード修正時に同期更新すべきドキュメント
ソースコードに変更を加えた場合は、作業完了前に必ず以下の該当ドキュメントを整合性がとれるよう更新すること（日本語版および英語版の双方を更新すること）。

- **`docs/ja/CHANGELOG.md` / `docs/en/CHANGELOG.md`**:
  変更内容や目的を最新セクションに自動追記する（ルートの `CHANGELOG.md` は案内リンクのみとする）。
- **`docs/ja/TODO.md` / `docs/en/TODO.md`**:
  着手したタスクや完了したタスクのステータス（`[ ]` ➔ `[x]`）を最新の状態に更新する。新規課題を発見した場合は Todo/Backlog に追記する。
- **`docs/ja/SPECIFICATION.md` / `docs/en/SPECIFICATION.md` (仕様書)**:
  新しい機能の追加や既存のデータ構造・画面仕様の変更があった場合に反映する。
- **`docs/ja/ARCHITECTURE.md` / `docs/en/ARCHITECTURE.md` (設計書)**:
  プロセスモデル、IPC通信、永続化設計、またはデータフローに変更があった場合に反映する。
- **`docs/ja/DEVELOPMENT.md` / `docs/en/DEVELOPMENT.md` (開発ガイド)**:
  ディレクトリ構成、依存関係、起動手順、外部連携等のルールに変更があった場合に反映する。
- **`docs/ja/RELEASE.md` / `docs/en/RELEASE.md` (リリース手順)**:
  ビルド・パッケージング手順、バージョンバンプフローに変更があった場合に反映する。
- **`docs/ja/USER_GUIDE.md` / `docs/en/USER_GUIDE.md` (ユーザーガイド)**:
  UI仕様や操作方法に変更があった場合に反映する。
- **`README.md` / `README_JA.md`**:
  環境構築や依存関係、ステータスバッジ（Shields.io）に影響する変更時に反映する。
- **`docs/ja/TEST_REPORT.md` / `docs/en/TEST_REPORT.md` (テスト結果報告)**:
  テスト実行時（自動/手動）に、検証手順や結果を記録・更新する。
- **`docs/ja/TESTING.md` / `docs/en/TESTING.md` (テスト方針・実行ガイド)**:
  テスト方針や検証環境の基準、ローカル検証プロセスに変更が生じた場合に反映する。
- **`docs/ja/FOOTPRINTS.md` / `docs/en/FOOTPRINTS.md` (フットプリント記録)**:
  リリースバイナリのフットプリント（ファイルサイズ、RAM使用量等）に変更が生じた場合に反映する。
- **`docs/ja/CONTRIBUTING.md` / `docs/en/CONTRIBUTING.md` (貢献ガイドライン)**:
  コミット規約、PR送信プロセス、セットアップ要件に変更が生じた場合に反映する。
- **`docs/ja/SECURITY.md` / `docs/en/SECURITY.md` (セキュリティポリシー)**:
  セキュリティ設計基準や脆弱性報告プロセスに変更が生じた場合に反映する。

## 8. 開発環境の統一とCI/CD
- **エディタ設定の厳守**:
  プロジェクトルートにある `.editorconfig` と `.vscode/settings.json` の設定を厳守すること。
  特に PowerShell スクリプト（`*.ps1`）は文字化けを防ぐため、必ず「BOM付き UTF-8」かつ「CRLF」で保存すること。その他のファイルは「BOMなし UTF-8」かつ「LF」で保存すること。
- **依存ライブラリの自動アップデート**:
  `.github/dependabot.yml` が設定されており、Cargo 依存関係および GitHub Actions のアップデートが自動で監視・提案される。
- **自動CI（継続的インテグレーション）**:
  リポジトリへのプッシュやプルリクエスト時に、GitHub Actions（`ci.yml`）によって自動的に `cargo fmt`、`cargo clippy`、`cargo test` が走るようになっている。ローカルでのコミット前にこれらが通ることを事前に確認すること。
  なお、無駄なビルドを防ぐため、Markdownファイル（`**/*.md`）のみの変更時はCIが起動しないよう設定（`paths-ignore`）されています。
- **手動リリース・自動デプロイフロー**:
  普段の Push では自動的にバージョンは上がりません。リリースする際は、ローカルで `scripts/bump-version.ps1` を実行してバージョンを更新・コミットし、バージョンタグ（例: `v1.3.5`）を作成してプッシュします。GitHub 側でタグのプッシュを検知すると、自動デプロイフロー（Release App）がトリガーされて、Windows 用の Tauri アプリがビルドされ GitHub Releases にドラフトリリースとしてデプロイされます。

## 9. 共有ライブラリの依存解決ルール
- **Cargo.toml の依存定義**:
  GitHub Actions や Dependabot 解析時にローカル相対パス参照エラー（`path_dependencies_not_reachable`）が発生するのを防ぐため、`common_lib` などの外部依存ライブラリは、必ず Git リポジトリ（`git = "..."`）として記述すること。
- **ローカルパスオーバーライド**:
  ローカル開発時に変更を迅速に反映させるため、`src-tauri/.cargo/config.toml` に `paths = ["../../common_lib"]` を指定してローカルパスを優先させる。この `config.toml` は開発環境依存となるため、必ず `.gitignore` に追加してリポジトリの管理対象外にすること。

## 10. ドキュメントコメント（RustDoc/JSDoc）の作成と同期
コードの追加や仕様変更時には、必ず対応するドキュメントコメント（RustDoc/JSDoc）を作成または更新すること。
- **Rust**: クレート、モジュール、構造体、列挙型、関数、メソッドなどに対して、`///` または `//!` を用いて RustDoc コメントを記述する。
- **JavaScript/TypeScript**: 各コンポーネント、カスタムフック、ユーティリティ関数などに対して、`/** ... */` を用いて JSDoc/TSDoc コメントを記述する。引数には `@param`、戻り値には `@returns` を適切に設定する。
- **言語設定**: コード内のコメントおよびドキュメントコメントは、英語表記を極力避け、日本語で統一すること。既存のコードで英語のコメントが存在する場合は、本プロジェクト開発時および仕様変更のタイミングで日本語へローカライズ（日本語化）すること。

## 11. 多言語ドキュメント管理ルール

プロジェクト内のドキュメント群の多言語対応を維持・管理するため、以下のルールを厳守すること。

- **フォルダによる言語管理の徹底**:
  - ドキュメントは必ず `docs/ja/` (日本語版) と `docs/en/` (英語版) の各フォルダ配下に配置すること。
  - 各種ドキュメントのファイル名に `_JA.md` や `.ja.md` などの言語サフィックスを付与することは禁止とする。
- **相互リンクの記述義務**:
  - 各言語のドキュメントの冒頭付近には、必ずもう一方の言語版へのリンクを記載すること。
  - 記述フォーマット例：
    - 日本語版： `[English Version](../en/FILENAME.md) | **日本語版**`
    - 英語版： `**English** | [日本語版](../ja/FILENAME.md)`
- **READMEの命名と相互リンク**:
  - プロジェクトルートのメインREADMEは英語版を `README.md` とし、日本語版は `README_JA.md` とする。
  - 双方の冒頭に以下の形式で相互リンクを記載すること。
    - `README.md`： `**English** | [日本語版](./README_JA.md)`
    - `README_JA.md`： `[English Version](./README.md) | **日本語版**`
- **CHANGELOGの構造**:
  - プロジェクトルートの `CHANGELOG.md` は、以下のように各言語版への案内リンクのみを記載したファイルとする。
    ```markdown
    # Changelog

    - [日本語版 (Japanese)](docs/ja/CHANGELOG.md)
    - [English Version](docs/en/CHANGELOG.md)
    ```
  - 実際の変更履歴は `docs/ja/CHANGELOG.md` および `docs/en/CHANGELOG.md` に記載し、これらにも冒頭に相互リンクを記述すること。

## 12. 品質管理・自動テストと事前検証ルール

AIエージェントがソースコードを変更した際、あるいはタスクを完了する前には、必ず以下のルールに従って品質管理と動作検証を行わなければなりません。

- **ユニットテストの追加・拡充**:
  - 新機能の追加やコアロジックの変更を行う場合、可能な限り関連するユニットテストコード（RustのバックエンドテストやフロントエンドのVitest等）を追加・拡張すること。
- **ローカルでの事前検証プロセス**:
  - コミットまたはタスクの完了報告を行う前に、ローカル環境で以下の検証コマンドを実行し、すべて合格（エラー・警告ゼロ）することを確認すること：
    1. `cargo test` （テストの合格）
    2. `cargo clippy --all-targets -- -D warnings` （Clippyの警告・エラーゼロ）
    3. `cargo fmt --check` （コードフォーマットの完全準拠）
    4. `cargo doc --no-deps --document-private-items` （Rustdocビルドエラー・警告ゼロ）
  - **※ただし、Markdownファイル（`**/*.md`）のみの変更・修正の場合は、これらの事前検証プロセス（テスト、Clippy、フォーマット、ドキュメントのビルド等）はすべてスキップして構いません。**
