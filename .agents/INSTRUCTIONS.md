# Clondar Pro AI 開発指示書 (INSTRUCTIONS.md)

本ドキュメントは、AIアシスタントが Clondar Pro のソースコード（Rust バックエンド / React フロントエンド）を修正・追加する際に厳守すべきコーディングスタイル、エラーハンドリング方針、コンポーネント設計、および出力フォーマットを定義したガイドラインです。

---

## 1. 命名規則

既存コードのスタイルを一貫して維持するため、以下の規則を厳守してください。

### 1.1 バックエンド (Rust)
- **ファイル名 / モジュール名**: `snake_case` (例: `main.rs`, `lib.rs`)
- **関数名 / 変数名**: `snake_case` (例: `load_holidays_json`, `get_holidays_file_path`, `app_handle`, `win_clone`)
- **構造体 / 列挙型 / 型名**: `CamelCase` (例: `QuittingState`, `DiffPart`)
- **定数**: `UPPER_SNAKE_CASE` (例: `DEFAULT_HOLIDAYS_JSON`)
- **Tauri コマンド名**: Rust側で定義するコマンド関数名は `snake_case` です。
  - ※ **重要**: 通常の Tauri アプリとは異なり、フロントエンドから呼び出す際（`invoke`時）もスネークケースをそのまま指定しています（例: `invoke('load_holidays_json')`）。JS側での自動キャメルケース変換を行わないように設定・連携されているため、呼び出し文字列のミスマッチに注意してください。

### 1.2 フロントエンド (Vite / React / JavaScript)
- **ファイル名**:
  - UIコンポーネントファイル: `PascalCase.jsx` (例: `Calendar.jsx`, `Clock.jsx`, `HolidaysManager.jsx`)
  - ユーティリティ / その他ファイル: `camelCase.js` (例: `holidays.js`, `tauri.js`)
- **コンポーネント関数名**: `PascalCase` (例: `WindowFrame`, `App`, `DigitalClock`)
- **一般関数名 / 変数名**: `camelCase` (例: `initHolidays`, `reloadHolidays`, `getHolidays`, `isRestoringRef`, `saved24H`)
- **定数**: `camelCase` または `UPPER_SNAKE_CASE` (例: `fallbackConfig`)

## 2. エラーハンドリング方針

システムがクラッシュしたり、画面がフリーズするのを防ぐため、各プロセスで例外・エラーを安全に処理・伝播させてください。

### 2.1 バックエンド (Rust)
- **原則**: `panic!` や `unwrap()` の使用は極力避け、回復可能なエラーは `Result<T, String>` などの戻り値で処理してください。
  - 起動プロセスでの「二重起動チェック（Named Mutexの取得失敗）」や「Tauri初期化失敗」のような致命的エラーの場合のみ、例外終了（`std::process::exit(0)` や `expect()`）を許容します。
- **IPC コマンドのエラー**:
  - フロントエンドに通知する Tauri コマンド関数は、エラー時に `Result<T, String>` を返すようにし、`.map_err(|e| e.to_string())` などを利用してエラー理由を文字列としてフロントエンドへ引き渡してください。

### 2.2 フロントエンド (React)
- **例外のキャッチ**:
  - Tauri API や Rust コマンドの呼び出し（`invoke`）、Web API（`fetch`、`JSON.parse`）などの失敗する可能性のある非同期処理は、必ず `try-catch` ブロックで保護してください。
- **フォールバック設計**:
  - ファイルの読み込みや通信エラーが発生した場合は、`console.error` でログを記録した上で、速やかに内蔵のデフォルト設定（`fallbackConfig`）を適用するなど、画面が崩れたり真っ白にならないように縮退（フォールバック）運転する設計にしてください。

## 3. コンポーネント・モジュール分割基準

コードの肥大化を防ぎ、保守性を高めるため、責任範囲に基づいてソースコードを分割してください。

### 3.1 バックエンド (Rust)
- **`main.rs` (メインプロセスロジック)**:
  - アプリケーションのセットアップ、ウィンドウ装飾（枠・影・透過）の制御、常駐トレイメニューの構築、およびIPCコマンドの登録を行います。
- **`common_lib` (共有クレート)**:
  - カレンダー処理、Diff計算、二重起動チェックなど、アルゴリズム的で別アプリケーションとも共有可能な純粋なロジックは、この独立したクレートに集約し、`main.rs` から呼び出す設計とします。

### 3.2 フロントエンド (React)
- **`ui/src/components/` (ビュー層)**:
  - UI描画とそれに密接に関連するローカル状態のみを処理する React コンポーネントを配置します（例: `Clock.jsx`）。
- **`ui/src/utils/` (ロジック・API層)**:
  - 描画ロジックを含まない、純粋なアルゴリズムや外部APIの抽象化を配置します（例: `holidays.js`, `tauri.js`）。
  - ビュー側が直接 `@tauri-apps/api` に依存するのを防ぐため、`tauri.js` でラッパー関数を定義し、非Tauri環境（ブラウザモック等）でのフォールバック処理をここに閉じ込めてください。
- **`App.jsx` (オーケストレーター)**:
  - アプリ全体の共通ステート（テーマ、設定値、バージョン等）を一元管理し、イベントのライフサイクル（移動イベント、トレイイベントの購読）を制御し、下位コンポーネントへProps経由でデータを流し込みます。

## 4. AIが出力する際のフォーマット指定

AIがコードの修正案をユーザーに提示する際は、迅速な確認と適用を可能にするため、以下のフォーマットを遵守してください。

### 4.1 解説の最小化
- コード変更時の前置きや過剰な解説は最小限に抑え、**修正コードそのものを主役にしてください**。
- 実装のポイントや注意点は、コードブロックの直後に簡潔な箇条書きで説明してください。

### 4.2 差分（Diff）の提示
- コードの修正を提案する際は、対象ファイル名、行番号、および明確な unified diff（`-` 削除行、`+` 追加行）を用いて、どの範囲に修正が加わるかを一目でわかるように提示してください。

### 4.3 コメントとドキュメントの日本語化
- 追加または編集するコード内のコメント（コードコメント、JSDoc/TSDoc、RustDoc）は、**すべて日本語で記述してください**。
- 既存コードの修正時に英語コメントを発見した場合は、そのタイミングで日本語に翻訳（ローカライズ）してください。

### 4.4 RustDoc/JSDoc の付与
- 新規追加するモジュール、構造体、関数、コンポーネントには必ず JSDoc や RustDoc を付与し、引数（`@param`）や戻り値（`@returns`）を記述してください。
